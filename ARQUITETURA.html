3. Fluxo de Autenticação

Toda a aplicação está protegida — sem login não se acede a nada

⚠️
Site totalmente protegido. O middleware interceta todos os requests e redireciona para /login se não houver sessão ativa. Inclui páginas públicas (homepage, imóveis, seguros, etc.).
📨 Request chega
Qualquer rota (exceto _next/static, imagens, favicon)
🛡️ Middleware — updateSession()
src/lib/supabase/middleware.ts
1. Cria Supabase client com cookie handler
2. supabase.auth.getUser() (refresh automático)
3. Cookies com maxAge: 2 horas (7200s)
✅ Tem sessão
Continua request normal
❌ Sem sessão
Redirect → /login?redirect=URL
📝 É /login
Permite acesso (excepção)

📝 LoginForm (Client Component)
src/app/login/login-form.tsx

7. Server Actions & API Routes

Endpoints do backend e como os dados fluem

⚡ Server Action: submitLead()

Ficheiro: src/app/actions/leads.ts

  • Chamado por <LeadForm> em contacto, visita imóvel, simulação seguro
  • Recebe FormData — extrai nome, email, telefone, mensagem
  • Mapeia campos dinâmicos com heurísticos PT (keywords: "matrícula", "capital", etc.)
  • INSERT na tabela leads via Supabase server client
  • revalidatePath('/admin/leads') para refresh imediato no admin
  • Retorna { success, error }

8. Padrão de Data Fetching

Como os dados chegam do Supabase aos componentes

📄 Server Component (padrão)
export const revalidate = 0

async function Page() {
  const supabase = await createClient()
  const { data } = await supabase
    .from('tabela').select('*')
  return <JSX />
}
⚡ Client Component (excepção)
"use client"

// Apenas para interatividade:
// - Formulários
// - Galerias de imagens
// - Mapas interativos
// - Theme toggle
// - Auth (login)
ℹ️
revalidate = 0 em tudo. Todas as páginas fazem SSR dinâmico — sem cache. Cada request faz uma query fresca ao Supabase. A performance depende do edge (Cloudflare) + latência ao Supabase.
ℹ️
Sem SWR, sem React Query, sem useEffect para dados. Os dados vêm sempre via React Server Components. Client components recebem dados como props.

13. Notas Importantes

Decisões arquiteturais e pontos de atenção

1

Site inteiro protegido por login — nenhuma página pública é acessível sem autenticação. Decisão intencional (fase beta/teste). Para abrir, remover o auth gate no middleware.

2

RLS (Row Level Security) desligado — está comentado no schema.sql. A segurança depende exclusivamente do middleware auth gate. Se o site for aberto ao público, considerar activar RLS.

3

Sessões de 2 horas — cookies expiram em 7200 segundos (hardcoded em middleware.ts e server.ts). Fix para compatibilidade com Safari iOS.

4

revalidate = 0 em toda a app — zero caching. Cada request faz uma query fresca ao Supabase. Para melhorar performance, considerar ISR com revalidate > 0.

5

Sem signup — utilizadores admin são criados manualmente no painel Supabase (Authentication → Users). Não existe formulário de registo.

6

form_fields dinâmicos nos seguros — cada seguro pode ter um formulário personalizado via JSONB. O <LeadForm> renderiza estes campos automaticamente.

7

Deploy manual — npm run deploy executado localmente. Sem CI/CD (GitHub Actions). Para automatizar, configurar workflow no .github/workflows/.

8

Fonts — Inter (corpo) + Outfit (headings) carregadas via next/font/google com subset latin. Optimizadas automaticamente pelo Next.js.
